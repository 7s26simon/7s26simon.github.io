CSP is a seatbelt, not a force field

A lab to demonstrate this is coming soon to: https://webverselabs-pro.com/

A look at a CSP Policy:

I was thinking about CSP and thought, what if.. there was a reasonable CSP:

script-src 'nonce-XXX'
frame-ancestors 'none'
base-uri 'none'

The script-src is the gold standard. No unsafe-inline, no unsafe-eval.
This is exactly what modern apps deploy. frame-ancestors prevents clickjacking. Solid. base-uri ‘none’ prevents <base> tag injection. Great so far.

But then, we see this:

connect-src * 
img-src 'self' data: *

Connect-src would be wide open. This configuration allows fetch() exfil to any origin. img-src ‘self’ data: * is wide open too. This allows a new Image().src to exfil anywhere.

For an attacker, this CSP policy tells them: “If I can execute JS, I have a free exfil channel.” The script-src nonce is the only wall. Everything now depends on whether they can break it.

script-src is the directive and it tells the browser which scripts are allowed to run.

'nonce-XXX' is the source value. It's a specific rule that says “only scripts with this exact nonce attribute can execute.”

Together, script-src 'nonce-XXX' mean that the browser will block every <script> tag on the page unless it has a matching nonce="XXX" attribute.

How it works in practice:

The server generates a random nonce per request and does two things with it:

  1. Puts it in the CSP header: script-src 'nonce-abc123'
  2. Puts it on every legitimate script tag: <script nonce="abc123">

An attacker who injects <script>alert(1)</script> gets blocked because their tag doesn't have the nonce. Additionally, they can't guess it because it's random and changes on every page load.

Creating a Lab (Concept)

The catch, and the whole point of this post, is that the nonce would be on the <script> element, not on individual lines of code inside it.

Important note: CSP validates the html layer, NOT inside a javascript <script> block.

Therefore if the script the attacker can inject is into an existing nonce’d script, their code would execute and run with full trust. The browser already approved that script block (via nonce, hash or allowed source).

The Code

So lets take this example, where we have a “callsign” input that a user can write into.

<script nonce="abc123">
  var operative = { callsign: "" + (ATTACKER CODE) + "" };
</script>

CSP checks: “Does this <script> have the right nonce?" Yes? Ok, let’s approve it! It never parses the JS inside to ask "wait, is that fetch() to a suspicious URL part of the original code or something injected?"

CSP is like a bouncer checking IDs at the door. Once you’re inside, it doesn’t watch what you do. If an attacker can smuggle their code inside a trusted script element, CSP has no opinion about it.

So something like this: "+alert(1)+" would fire because it gets injected into the script block:

<script>
      var operative = {
        callsign: ""+alert(1)+"",
        report_id: 1,
        viewed_at: new Date().toISOString()
      };
      // Banner update — purely cosmetic.
      try {
        document.getElementById('callsign-display').textContent = operative.callsign;
      } catch (e) {}
</script>

Lessons Learned

I hope this demonstrates why a strong CSP policy can be bypassed and is a defence in depth layer approach. Input sanitization would be the real fix here. The server should JSON.stringify() the callsign before inserting the user controlled field data into the <script> block.

Without JSON.stringify():

callsign: ""+alert(1)+""

The " breaks out of the string and the payload is now executed.

With JSON.stringify():

callsign: "\"+alert(1)+\""

The quotes are escaped with backslashes.

CSP is a seatbelt, not a force field. It reduces blast radius but doesn’t fix the underlying bug. If your nonce’d scripts contain unsanitized user input, your nonces are protecting the attacker’s code just as well as yours.
